# EcoVault 开发规范

## 基本原则

- 遵循《阿里巴巴 Java 开发手册》中的命名、异常、集合、并发、日志与数据库规范。
- 所有注释、文档和面向用户的说明使用中文。
- 所有文件统一使用 UTF-8 编码。
- Java 基础包名统一为 `com.tlcsdm.ecovault`。
- 每次功能、接口、数据库、安全策略、部署流程变更，都必须同步更新 `docs/` 文档。

## 命名规范

- 类名使用大驼峰，例如 `PasswordEntryService`。
- 方法名、变量名使用小驼峰，例如 `createPasswordEntry`。
- 常量使用全大写下划线，例如 `MAX_DEVICE_COUNT`。
- 数据库表名和字段名使用小写下划线，例如 `password_entries`、`created_at`。
- RESTful 路径使用小写复数名词，例如 `/api/passwords`。

## 包结构规范

```text
com.tlcsdm.ecovault
├── aspect
├── config
├── controller
├── dto
├── entity
├── exception
├── interceptor
├── repository
├── security
├── service
└── utils
```

## 中文注释要求

- 复杂业务逻辑、安全策略、加密流程、数据转换必须添加中文注释。
- 注释应解释“为什么这样做”，避免重复描述显而易见的代码。
- 对外 API、配置项、数据库字段需要有清晰中文说明。

## Git 提交规范

提交信息建议格式：

```text
<类型>: <中文简短说明>
```

常用类型：`feat`、`fix`、`docs`、`test`、`refactor`、`build`、`ci`。

示例：

```text
feat: 增加密码标签筛选功能
```

## 分支命名规范

分支名必须以用户名开头：

```text
<用户名>/<类型>/<简短主题>
```

示例：

```text
unknowIfGuestInDream/feat/password-tags
unknowIfGuestInDream/fix/login-session
```

## RESTful API 规范

- 使用名词复数表示资源，例如 `/api/finance/salaries`。
- 使用 HTTP 方法表达动作：`GET` 查询、`POST` 创建、`PUT` 全量更新、`PATCH` 局部更新、`DELETE` 删除。
- 响应结构应统一包含状态码、消息、数据与追踪 ID。
- 分页参数统一使用 `page`、`size`、`sort`。
- 禁止在 URL 中传递敏感信息。

## 异常处理规范

- 使用全局异常处理器统一返回错误响应。
- 业务异常应使用明确错误码与中文错误消息。
- 安全异常不得暴露认证细节。
- 日志中必须脱敏敏感字段。
- 禁止吞掉异常后不记录或不返回明确结果。

## 安全规范

- 用户密码必须使用 BCrypt 哈希存储。
- 密码条目敏感字段必须使用 AES 加密存储。
- JWT 密钥、AES 密钥、数据库路径等敏感配置必须从配置文件或环境变量读取。
- JWT 默认登录时长为 2 小时，必须通过 `ecovault.security.jwt-expiration-ms` 统一控制令牌与 Cookie 过期时间。
- ADMIN 才能访问 Actuator 详细信息与管理后台；操作日志仅管理员可查看，普通用户不可访问日志接口与页面。
- 页面级访问权限通过角色管理配置，服务端须二次校验，禁止仅依赖前端菜单隐藏。
- 所有数据库查询必须使用参数绑定，禁止拼接 SQL。
- 前端输出必须防止 XSS，表单接口必须考虑 CSRF 防护。

## 单元测试要求

- 新增业务逻辑必须添加或更新 JUnit 5 测试。
- 安全、加密、权限、导出、统计等核心逻辑必须覆盖边界场景。
- 修复缺陷时应优先补充失败用例，再修复实现。
- 本地提交前至少运行：

```bash
mvn test
```

- JaCoCo 报告位置：`target/site/jacoco/index.html`。
- GitHub Actions 需要能稳定生成覆盖率摘要，禁止在工作流脚本中引入会导致默认 `awk` 解析失败的写法。
- 每次代码修改都必须关注单体测试分支覆盖率，新增分支逻辑时应补齐对应的正反向与异常路径用例。

## 文档更新要求

以下变更必须同步更新文档：

- 新增或调整功能模块。
- 新增、删除或修改 API。
- 修改数据库表、字段、索引或约束。
- 修改认证、授权、加密或审计策略。
- 修改构建、部署、Docker、Jenkins 或运行配置。
- 调整 GitHub Actions 触发条件或跳过规则。

## 敏感信息要求

- 禁止提交真实密钥、Token、密码、数据库文件、私有证书或生产配置。
- Pull Request 必须确认“无敏感信息”。
- 示例配置只能使用占位符，不得使用真实凭据。
